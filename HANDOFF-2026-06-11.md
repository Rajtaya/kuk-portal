### 38. Cross-origin cookie fix for Railway deployment

The Session 13 security migration (Bearer ‚Üí HttpOnly cookie auth) broke login on Railway because frontend (`uems.up.railway.app`) and backend (`backend-production-7615.up.railway.app`) are on different subdomains. Cross-origin cookies require `SameSite=None; Secure`.

**Problem:** Multiple attempts to detect "cross-origin" via env vars (`CORS_ORIGIN`, `CORS_ORIGINS`, `RAILWAY_ENVIRONMENT`) all failed ‚Äî Railway wasn't picking up the env var checks at module load time or request time.

**Solution:** Hardcoded `SameSite=None; Secure=true` always. This works everywhere:
- **Railway (HTTPS, cross-origin):** correct ‚Äî browser sends cookie cross-origin
- **Localhost (HTTP):** Chrome/Firefox accept `Secure` cookies on localhost since ~2020

**Commits pushed:** 6 iterations (`6978be6` ‚Üí `4e54000` ‚Üí `b3f48b5` ‚Üí `f108930` ‚Üí `1c14c9e` ‚Üí `f16a52f`), final working version is `f16a52f`.

**Railway deploy gotcha:** Auto-deploy from GitHub webhook was unreliable ‚Äî multiple pushes didn't trigger deploys. Empty commits (`git commit --allow-empty`) sometimes forced it. Check Deployments tab if pushes don't auto-deploy.

- Files: `backend/src/auth/auth.controller.ts`

## What was done in Session 13 ‚Äî 2026-06-11

### 37. Security hardening ‚Äî Copilot audit remediation (7 findings, all addressed)

Triggered by a GitHub Copilot security audit. All 7 findings fixed in one session.

#### Fix 1: IDOR on document listing (High ‚Üí Fixed)
- `GET /documents/:employeeId` had **no authorization check** ‚Äî any authenticated user could list any employee's documents
- **Fix:** added `@Roles(SUPER_ADMIN, STATE_USER, UNIVERSITY_ADMIN)` + ownership verification for UNIVERSITY_ADMIN via `verifyEmployeeOwnership()`
- Files: `documents.controller.ts`

#### Fix 2: File upload unrestricted (Medium ‚Üí Fixed)
- Upload accepted any file type with only a size limit; served from same origin
- **Fix:** MIME allowlist (`application/pdf`, `image/jpeg`, `image/png`, `image/webp`) in Multer `fileFilter` + server-side extension validation in `documents.service.ts`
- Files: `documents.controller.ts`, `documents.service.ts`

#### Fix 3: CORS fails open in production (Medium ‚Üí Fixed)
- When no `CORS_ORIGIN` env var set, `origin: true` reflected all origins
- **Fix:** production without `CORS_ORIGIN` now defaults to `origin: false` (deny all) with a console warning
- Files: `main.ts`

#### Fix 4: Seed script runs in production (Medium ‚Üí Fixed)
- `prisma/seed.ts` would happily run with `admin123` passwords in production
- **Fix:** hard exit if `NODE_ENV=production` + console warning about default passwords
- Files: `seed.ts`

#### Fix 5: Uploads served publicly without auth (Medium ‚Üí Fixed)
- `/uploads/` directory served as public static files ‚Äî documents fetchable if URL known
- **Fix:** only `/uploads/photos/` served statically (employee photos, with `X-Content-Type-Options: nosniff` + `Content-Security-Policy: default-src 'none'`). Documents served through new authenticated `GET /documents/download/:id` endpoint with ownership check + `Content-Disposition: attachment`
- **Bonus:** employee photo upload extension now derived from validated MIME type, not user-supplied filename
- Files: `main.ts`, `documents.controller.ts`, `documents.service.ts`, `employees.controller.ts`

#### Fix 6: JWT stored in localStorage ‚Äî XSS token theft (High ‚Üí Fixed)
- Bearer token was persisted in `localStorage` and attached via `Authorization` header ‚Äî any XSS could exfiltrate it
- **Fix:** full migration to **HttpOnly cookie auth**:
  - Backend: `cookie-parser` middleware, login sets `auth_token` cookie (HttpOnly, Secure, SameSite=None, 24h expiry), new `POST /auth/logout` clears cookie
  - JWT strategy: extracts token from cookie first, falls back to `Authorization` header (API clients still work)
  - Frontend: removed ALL `localStorage` token code, all `fetch()` calls use `credentials: 'include'`, auth-context checks session via `GET /auth/profile` on load
  - Fixed 2 direct `localStorage.getItem('token')` calls in employee template download functions
- Files: `auth.controller.ts`, `jwt.strategy.ts`, `main.ts`, `api.ts`, `auth-context.tsx`, `employees/page.tsx`, `employees/upload/page.tsx`

#### Fix 7: No account lockout (Low/Medium ‚Üí Fixed)
- Only had basic throttling (5 req/60s) ‚Äî no lockout after repeated failures
- **Fix:** in-memory attempt tracker in `auth.service.ts`:
  - **5 failed attempts ‚Üí 15-minute account lockout**
  - Shows remaining attempts on failure ("Invalid credentials. 3 attempt(s) remaining.")
  - Shows lockout duration ("Account temporarily locked. Try again in 14 minute(s).")
  - Clears on successful login
  - No DB schema change (in-memory Map, resets on server restart ‚Äî acceptable for single-instance deployment)
- Files: `auth.service.ts`

#### Security posture summary (after all fixes)
| Before | After |
|--------|-------|
| XSS ‚Üí token theft | HttpOnly cookie, JS cannot access token |
| Any user ‚Üí any employee's documents | Role + university ownership check |
| Upload .html/.svg ‚Üí stored XSS | PDF/JPEG/PNG/WebP only, nosniff headers |
| CORS allows all origins in prod | Deny-all unless explicit allowlist |
| Seed runs anywhere | Blocked in production |
| Brute force: 5 req/min throttle only | Throttle + 5-attempt lockout (15 min) |
| Documents publicly fetchable by URL | Authenticated download endpoint |

## Key Gotchas
- **Prisma v6 pinned:** Global Prisma is v7, project pins v6. Always use `./node_modules/.bin/prisma`, never bare `npx prisma`. Use `prisma db push` (not `prisma migrate dev`) due to DB drift
- **NestJS build path:** Output goes to `dist/src/main.js`, not `dist/main.js`
- **Railway monorepo:** Uses `RAILWAY_DOCKERFILE_PATH=backend/Dockerfile` env var. Dockerfiles use `COPY backend/` (project root context)
- **Railway watches `Rajtaya/UEMS`** repo (not `kuk-portal`). Push goes to both repos via multi-URL origin config
- **Railway auto-deploy flaky:** webhook sometimes doesn't fire. Try empty commit push or manual redeploy from dashboard
- **Next.js cache:** Sometimes needs `.next/` deleted + server restart to pick up changes
- **Frontend env for Railway:** `NEXT_PUBLIC_API_URL` must be set as Docker build arg (baked at build time, not runtime). Also set in `.env.production`
- **Deploy:** `git push` to `main` auto-deploys both services on Railway. Push goes to both `kuk-portal` and `UEMS` repos automatically
- **Railway CDN cache:** Root layout has `export const dynamic = 'force-dynamic'` + `Cache-Control: no-cache` in next.config.ts. Don't remove these
- **Dockerfile must copy public/:** standalone Next.js build does NOT auto-include `public/`. Dockerfile has `COPY --from=builder /app/public ./public`
- **Frontend dev:** port 3001, backend port 4000. Backend start: `npm run dev` (NOT `npm run start:dev`)
- **Login automation:** use `admin@he.haryana.gov.in` / `admin123` for SUPER_ADMIN, `admin@mdu.ac.in` / `admin123` for UNIVERSITY_ADMIN, `state@he.haryana.gov.in` / `admin123` for STATE_USER. Note: 5 failed attempts trigger 15-min lockout (resets on backend restart)
- **Auth is cookie-based:** `SameSite=None; Secure=true` always (works on localhost and Railway). Frontend uses `credentials: 'include'` on all fetches. JWT is in an HttpOnly cookie (`auth_token`), NOT in localStorage. JWT strategy still accepts `Authorization: Bearer` header as fallback for API testing (Swagger/curl)
- **Document downloads:** documents are NOT served from `/uploads/` anymore. Use `GET /api/documents/download/:id` (authenticated). Only `/uploads/photos/` is still static (employee photos)
- **File uploads:** only PDF, JPEG, PNG, WebP accepted. Both MIME and extension validated server-side
- **CORS in production:** requires `CORS_ORIGIN` or `CORS_ORIGINS` env var. Without it, production denies all origins
- **Seed script:** refuses to run if `NODE_ENV=production`
- **DB sync:** `pg_dump` local ‚Üí `pg_restore` to Railway. Always use `--clean --if-exists --no-owner --no-acl`
- **Never add Co-Authored-By** lines to git commits (user preference)
